# x-ui-node-limit-manager

一个适用于 Debian 的 x-ui / 3x-ui 节点限速管理脚本。  
用于对 **x-ui 面板中的独立端口节点** 进行单独限速，支持交互式操作、开机自动恢复、规则查看、修改、删除与修复。

> 适合场景：  
> 一台 Debian 服务器，安装了 x-ui / 3x-ui 面板，希望对其中某一个或多个节点按端口单独限速，而不是对整台服务器或全部节点统一限速。

---

## 功能特点

- 交互式菜单操作
- 自动扫描服务器网卡并按编号选择
- 自动检测 x-ui / 3x-ui 数据库
- 自动读取 x-ui 面板节点并按编号选择
- 默认优先显示已启用节点
- 支持为单个节点按端口限速
- 支持协议选择：
  - 仅 TCP
  - 仅 UDP
  - TCP + UDP
- 支持：
  - 添加限速
  - 删除限速
  - 修改限速
  - 查看当前状态
  - 清空某张网卡上的全部限速
  - 修复 / 清理残留规则
- 支持开机自动恢复当前限速规则
- 输出简洁状态视图，便于快速判断是否生效

---

## 原理说明

本脚本的限速方式基于：

- `iptables` / `mangle`
- `tc` / `htb`

基本流程为：

1. 对指定节点端口的流量打上 `fwmark`
2. 使用 `tc` 将该标记流量归入指定的限速 class
3. 实现对单个节点端口的独立带宽限制

### 当前脚本限制的方向

当前脚本主要限制的是：

- **服务器出站流量**

也就是从用户体验看，通常表现为：

- **限制客户端通过该节点的下载速度**

---

## 适用条件

### 推荐使用场景

- x-ui / 3x-ui 中每个节点使用**独立监听端口**
- 例如：
  - `50000`
  - `57331`
  - `45720`
  - `12106`

### 不推荐场景

如果多个节点：

- 共用同一个监听端口
- 依赖 fallback / path / SNI / 其他方式区分

那么脚本按端口限速的方式就不适用了，或者无法准确只限某一个节点。

## 系统要求

- Debian / Debian 系发行版
- 需要 root 权限
- 已安装并正常使用 x-ui / 3x-ui
- 系统具备：
  - `iptables`
  - `iproute2`
  - `sqlite3`

脚本可通过菜单自动安装依赖。

## 使用说明

### 1. 添加节点限速

选择：

```text
2) 添加节点限速
```

脚本会依次执行：

1. 扫描当前机器网卡
2. 自动检测 x-ui 数据库
3. 列出 x-ui 面板中的节点
4. 选择要限速的节点
5. 输入限速值（单位 Mbps）
6. 选择协议类型
7. 确认后写入限速规则
8. 可选择是否启用开机自动恢复当前限速规则

#### 协议选择建议

- `VLESS + TCP + Reality` 一般选择：
  - **仅 TCP**
- `tunnel` 或不确定类型时，可选择：
  - **TCP + UDP**

---

### 2. 查看当前状态

选择：

```text
3) 查看当前状态
```

显示的是简洁视图，通常包含：

- 网卡
- 端口
- 协议
- 速率
- 加载状态
- 流量命中

#### 加载状态说明

- `已加载`：说明 iptables 与 tc 规则链路完整
- `异常`：说明规则记录存在，但底层限速链路不完整或有异常

#### 流量命中说明

- `有`：说明该规则最近命中过流量
- `暂无`：说明规则存在，但暂时没有流量经过

---

### 3. 删除节点限速

选择：

```text
4) 删除节点限速
```

删除时会优先列出：

- 当前**已经配置的限速规则**

而不是重新从全部 x-ui 节点中选择。

可选删除方式：

- 删除该节点的全部限速
- 仅删除指定协议的限速

---

### 4. 修改节点限速

选择：

```text
5) 修改节点限速
```

用于直接修改某个已限速节点的速率，避免手动删除后再重加。

---

### 5. 清空某网卡全部限速

选择：

```text
6) 清空某网卡全部限速
```

会删除该网卡上由脚本创建的全部限速规则。

---

### 6. 修复 / 清理残留规则

选择：

```text
7) 修复/清理残留规则
```

适用于以下情况：

- 之前反复测试过
- 有旧规则残留
- `tc filter` / `tc class` 状态混乱
- 显示“有命中”但限速不生效
- `rules.conf` 与当前实际规则不一致

该功能会：

1. 去重规则文件
2. 删除脚本规则对应的 iptables 标记
3. 清理相关网卡上的 tc root qdisc
4. 按当前规则文件重新加载限速规则

---

### 7. 开机自动恢复当前限速规则

选择：

```text
8) 开机自启设置
```

可启用或关闭：

- **开机自动恢复当前限速规则**

含义是：

> 服务器重启后，自动重新加载当前已保存的限速规则。

不是“恢复成系统原样”，而是恢复成脚本中当前已保存的限速配置。

---

### 8. 环境检查

选择：

```text
9) 环境检查
```

可快速检查以下内容：

- 是否为 root 权限
- `iptables` 是否可用
- `tc` 是否可用
- `sqlite3` 是否可用
- 是否检测到 `x-ui.db`
- 开机自动恢复功能是否已启用

## 注意事项

### 1. 本脚本按“端口”限速

如果一个节点独占一个端口，那么就等于实现了：

- 对单个节点限速

但如果多个节点共用同一端口，则无法精准区分。

---

### 2. 限速的是该端口总带宽

例如：

- 对 `57331` 设置 `30mbit`

表示：

- 该端口的总带宽上限约为 `30Mbit`

不是每个客户端都各自拥有 `30Mbit`。

---

### 3. 当前主要限制服务器出站方向

也就是客户端通过该节点的“下载速度”。

如需同时精细处理更复杂的双向流量整形，需要进一步扩展为 ingress / ifb 方案。

---

### 4. 部分节点类型需自行判断协议

例如 `tunnel` 类型，可能涉及 TCP / UDP / 两者都有。  
如果不确定，建议优先选择：

- `TCP + UDP`

---

### 5. 旧规则残留问题

如果在开发、调试或多次修改过程中出现规则异常，建议执行：

```text
7) 修复/清理残留规则
```

## 常见问题

### Q1：安装 iptables 后，需要重新一个个放行端口吗？

**不需要。**

本脚本使用的是：

- `iptables -t mangle`

主要用于打标记，不是默认防火墙拦截策略。  
不会因为脚本本身而把所有端口都封掉。

---

### Q2：为什么“流量命中=有”，但“加载状态=异常”？

这通常表示：

- iptables 规则命中了
- 但 tc class / filter 没有完整建立

可以尝试：

```text
7) 修复/清理残留规则
```

---

### Q3：为什么某些端口以前添加失败？

较早版本脚本可能会使用过大的 `classid / handle`，导致：

- `invalid class ID`

当前版本已修复该问题，使用了更安全的 ID 范围。

---

### Q4：脚本能否自动读取 x-ui 节点？

可以。  
脚本会自动尝试查找：

- `x-ui.db`

并从数据库中读取节点信息。  
如果读取失败，会回退为手动输入端口模式。

## 兼容性说明

本脚本已在以下思路上设计：

- 适合 Debian
- 适合 x-ui / 3x-ui
- 适合独立端口入站节点

但不同版本的：

- x-ui
- 3x-ui
- xray-core
- 系统 iptables / nftables 兼容层
- 网卡驱动 / qdisc 环境

都可能存在差异。

因此如遇到特殊环境问题，建议结合以下命令排查：

```bash
iptables -t mangle -L OUTPUT -n -v
tc class show dev <网卡名>
tc filter show dev <网卡名> parent 1:
tc -s class show dev <网卡名>
```
